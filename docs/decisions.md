# DECISIONS.md

## 1. Tech Stack

### Python + Pandas

The core of the problem is data processing rather than application logic. Most of the effort goes into cleaning inconsistent records, handling edge cases, and performing reconciliation.

Pandas was a natural choice because it makes it easy to:

* iterate quickly on messy datasets
* perform joins and aggregations
* inspect intermediate results while debugging

Python also made it straightforward to prototype and validate assumptions early.

---

### FastAPI

FastAPI is used as a thin layer to expose processed data to the frontend.

It was chosen because it is:

* simple to set up
* fast enough for local data serving
* well-suited for returning structured JSON

The API does not contain business logic. It only serves outputs generated by the pipeline.

---

### React + Vite + Tailwind

A lightweight frontend was added to make the results easier to explore.

* React is used for structuring the UI
* Vite keeps the development loop fast
* Tailwind helps keep styling simple and consistent

The goal of the UI is not completeness, but clarity. It allows:

* quick filtering of workers
* inspection of discrepancies
* shift-level drilldown when needed

---

### Local-first setup

The system runs entirely locally:

* no database
* no external dependencies

This keeps setup simple and makes it easy to reproduce results. The focus stays on correctness rather than infrastructure.

---

## 2. Pipeline Structure

The pipeline is split into multiple scripts:

* `clean_data.py`
* `match_workers.py`
* `production_pipeline.py`
* `reconcile.py`
* `financial_audit.py`

Each stage has a clear responsibility. This made it easier to:

* debug issues in isolation
* validate intermediate outputs
* avoid tightly coupled logic

---

## 3. Worker Matching

### Primary: phone number

Phone number is used as the primary identifier because it is the most stable field across datasets.

Names are inconsistent and often unreliable.

---

### Fallback: name matching

When phone matching fails, a simple token-based name match is used with a threshold.

This is intentionally conservative.

---

### Tradeoff

The matching strategy favors precision over recall.

Ambiguous cases are flagged instead of being forced into a match. This avoids introducing silent errors into later stages.

---

## 4. Wage Rate Logic

Expected pay is computed using:

* role
* state
* seniority
* effective date

Rates are selected based on the shift date.

---

### Overlapping rates

If multiple wage rates match a single shift, the system does not try to resolve it automatically.

Instead, the shift is flagged.

Choosing a rate arbitrarily would hide a data issue. It is safer to surface ambiguity.

---

## 5. Reconciliation

### Worker-level aggregation

Reconciliation is done at the worker level.

Bank transfers do not reliably map 1:1 to shifts, so aggregating reduces noise and makes patterns easier to see.

---

### Tolerance

A tolerance of ₹100 is used when comparing expected and actual totals.

This avoids flagging small differences caused by rounding or minor inconsistencies.

---

## 6. Audit Layer

A separate audit step breaks the discrepancy into components:

* missing payments
* incorrect amounts
* corrupted transfers

It also includes a simple simulation to estimate the impact of fixing missing payments.

This step was useful to validate that the main hypothesis actually explains the observed gap.

---

## 7. Frontend Data Handling

The frontend does not recompute business logic.

It displays results produced by the pipeline and derives only simple aggregates needed for presentation.

Global metrics are computed from the same datasets used in validation to ensure consistency with the backend results.

---

## 8. Tradeoffs and Limitations

### Wage rate overlaps are not resolved

These cases are flagged rather than automatically handled.

This leaves some records unresolved, but avoids incorrect assumptions.

---

### UI is intentionally minimal

The interface is functional but not fully polished.

Time was spent on correctness of the pipeline rather than building a complete product experience.

---

### Phone number as unique identifier

This assumption simplifies matching but may not always hold (e.g., shared numbers or changes over time).

Ambiguous cases are flagged where possible.

---

## 9. What I Would Do Next

* Introduce a database (e.g., PostgreSQL) to manage larger datasets
* Track historical worker identities instead of relying on current state
* Add better observability for failed or missing payments
* Build a more complete UI for non-technical users
* Integrate validation checks earlier in the data pipeline

---

## 10. Summary

The system is designed to be explicit about uncertainty.

Where the data is ambiguous, it is surfaced rather than resolved silently. The focus throughout is on producing results that are easy to verify and reason about.
